# 기획서: 영수증 전표 자동화 (가칭 "찍으면 끝")

> ⚠️ **SUPERSEDED — 원 해커톤 기획(역사적 참고).** 현재 기준선(Source of Truth)은 [`sot/`](../sot/README.md). 목적·스코프는 [`sot/01_project_purpose.md`](../sot/01_project_purpose.md)가 원본이며, 충돌 시 sot/ 가 우선합니다.

> 10시간 해커톤 · 5인 팀 · 모바일 웹앱 + Mock E-Accounting 시연

## 0. PM의 첫 번째 결정: "데모에서 역산한다"

10시간짜리 프로젝트의 기획은 기능 목록에서 시작하면 망합니다.
**최종 3분 데모 시나리오를 먼저 확정하고, 그 시나리오에 등장하는 화면과 데이터 흐름만 만듭니다.**
데모 스크립트가 곧 스펙입니다. (시나리오는 아래 6장)

## 1. 문제 정의 (As-Is)

현재 출장비 정산 프로세스:

1. 출장을 간다
2. 현지에서 법인카드로 결제한다
3. 영수증을 잘 챙겨둔다
4. 사진을 찍거나 스캔한다
5. 잘 보이게 크롭한다
6. E-Accounting에서 해당 카드내역을 골라 전표작성 화면으로 넘어간다
7. 한 땀 한 땀 line-by-line으로 영수증을 첨부한다
8. 전결규정에 맞춰 결재를 올린다

### 핵심 페인포인트

| # | 페인포인트 | 원인 |
|---|-----------|------|
| P1 | 영수증 촬영→크롭→건별 첨부가 전부 수작업 | 자동화 부재 |
| P2 | 카드내역 리스트가 너무 많아 어떤 게 내 건지 헷갈림 | 매칭 수단 없음 |
| P3 | 건마다 전결라인을 다르게 잡아야 함 (복지비/도서비/AI구독 등) | 전결규정이 머릿속에만 있음 |
| P4 | 복지비 등 잔여 한도를 알 수 없음 | 예산 조회 기능 부재 |
| P5 | 출장 일일한도(Cap)를 인원수·환율 반영해 수동 계산 | 실시간 계산 도구 없음 |

## 2. 목표 (To-Be) — 핵심 가치 3가지

1. **찍으면 끝**: 영수증 사진 한 장 → 자동 크롭 + OCR + 카드내역 자동 매칭
2. **전표 자동완성**: 계정과목·전결라인이 자동으로 잡힌 전표 초안 생성. 사람은 **눈으로 검토하고 상신 버튼만** 누른다
3. **실시간 한도**: 출장모드에서 환율 반영 잔여 Cap, 복지비 잔여 예산을 즉시 확인

## 3. MVP 범위

### 반드시 만든다 (데모에 나옴)

- 모바일 웹앱: 영수증 촬영 → 자동 크롭(간단 처리) → OCR 결과 확인
- 출장모드: 출장 등록(국가/기간/인원) → 결제 시 환율 계산 → 일일 잔여 Cap 표시
- 자동 매칭: 영수증 ↔ Mock e-accounting 카드승인내역 (금액/일시/가맹점 기반)
- 전표 자동생성: 계정과목 자동분류 + 전결라인 자동 결정 → 검토 화면 → 상신
- Mock E-Accounting 웹화면: 카드내역 리스트, 접수된 전표와 결재라인 표시 (시연용)
- 복지비 한도: 카테고리별(복지/도서/AI구독) 잔여 예산 표시 및 차감

### 잘라낸다 (10시간에 안 됨 — 데모에서 말로만 언급)

- 로그인/권한 관리 (사용자 1명 하드코딩)
- 실제 결재 진행 프로세스 (결재라인 "표시"까지만, 승인 클릭은 생략 또는 버튼만)
- 네이티브 앱 스토어 배포 (모바일 웹/PWA로 대체)
- 실시간 환율 API 연동 실패 대비 → 고정 환율 테이블 fallback
- 푸시 알림, 다국어, 영수증 위변조 검증

## 4. 핵심 아키텍처 결정 (PM이 첫 1시간에 못박을 것)

| 결정 | 선택 | 이유 |
|------|------|------|
| 모바일 "앱" | **모바일 웹(PWA)**, 네이티브 X | 10시간 내 배포·시연 리스크 최소. 카메라는 `<input type="file" capture="environment">`로 충분 |
| OCR + 파싱 | **Vision LLM API 한 방** (Claude API 등) | OCR→금액/통화/가맹점/일시 구조화 추출을 프롬프트 하나로. 전통 OCR+파서 조합보다 빠르고 견고 |
| 자동 크롭 | 클라이언트 간단 처리 or LLM에 위임, 실패 시 원본 사용 | 데모에서 "크롭된 것처럼" 보이면 충분. 여기에 시간 쓰지 말 것 |
| Mock E-Accounting | 별도 REST 서버 + 간단한 관리자 웹화면 | 사내망 연결 불가 → 시연의 "회사 시스템 화면" 역할 |
| 통합 방식 | **첫 1시간에 API 계약 + JSON fixtures 고정** | 5명이 서로 안 기다리고 병렬 개발하는 유일한 방법 (02-API-CONTRACT.md) |
| 환율 | 고정 테이블 (USD/JPY/EUR 등 5개) | 데모에 실시간성 불필요. 여유 되면 무료 API |

## 5. 사용자 여정 (To-Be)

### 출장 중
1. 앱에서 "출장모드 시작" (도쿄, 3박4일, 2명 → 일일 Cap 자동 계산)
2. 저녁식사 후 영수증 촬영 → 자동 크롭 → ¥8,800 인식 → 원화 환산 → **"오늘 잔여한도 ₩23,000"** 즉시 표시
3. 영수증은 앱에 자동 보관 (분실 걱정 끝)

### 귀국 후 정산 (월말)
4. "정산하기" 탭 → 보관된 영수증들이 카드승인내역과 **자동 매칭** (매칭 스코어 낮은 건만 사용자 확인)
5. "전표 생성" → 계정과목·전결라인이 자동으로 잡힌 전표 초안
6. 사용자는 검토 후 "상신" → Mock E-Accounting에 전표 접수, 결재라인 태워짐

### 일상 복지비
7. AI 구독료 결제 → 앱이 "복지비-AI구독" 분류 제안 + **잔여 한도 표시** → 전결라인 자동 (한도 내 소액은 팀장 전결)
