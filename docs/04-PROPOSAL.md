# 기획서: 법인카드 정산 고도화 — 정산단위 기반 정산엔진

> 10시간·5인 해커톤 MVP(`01-PLANNING.md`)를 기반으로 한 고도화 PoC 기획서.
> 상세 설계·데이터 모델·API·시나리오는 [`../sot/`](../sot/README.md) 폴더를 참고 (이 문서가 "왜", SoT가 "어떻게").

## 1. 한 줄 요약

법인카드 정산에서 반복되는 두 가지 문제 — **① 증빙 처리가 전부 수작업**이라는 것과 **② 정산 규정마다 사람이 매번 기억하고 안내해야 한다**는 것 — 을 하나의 구조로 함께 푼다. 영수증은 찍으면 자동으로 처리되고, 정산 규정은 **정산단위**이라는 형태로 한 번 배포하면 이후 정산이 코드 변경 없이 그 규정을 그대로 따라간다.

## 2. 문제 정의 (As-Is)

### 2.1 증빙 처리가 전부 수작업이다

현재 출장비 등 법인카드 정산은 촬영 → 크롭 → E-Accounting에서 카드내역 찾기 → 건별 첨부 → 전결규정에 맞춰 결재까지 8단계를 사람이 손으로 처리한다 (`01-PLANNING.md` 1장). 영수증을 잃어버리거나, 카드내역 리스트에서 내 건을 못 찾거나, 전결라인을 매번 다시 확인해야 하는 비효율이 반복된다.

### 2.2 정산 규정이 "그때그때 안내"로 관리된다

법인카드 정산에는 두 종류의 규정이 있다.

- **표준 규정**: 카테고리별 금액구간에 따른 전결라인 (기존에도 시스템에 있음).
- **비표준 규정**: 출장비 한도(지역·기간에 따라 달라짐), 특수 목적 예산(AI Tool 구독료, 신규 구성원 지원, 팀빌딩 등) — 월 한도·예산 항목·전결 Line·적요가 미리 정해져 있지만, 이게 시스템에 구조화되어 있지 않고 **메일로 안내하고 사람이 기억해서 수동으로 정산 규칙에 맞춰야 하는** 방식으로 운영된다.

새로운 비표준 규정이 생길 때마다 안내 → 기억 → 수동 categorize라는 비용이 반복되고, 규정을 안내받은 사람이 잊어버리면 정산이 틀어진다.

### 2.3 부수적 리스크: 적격증빙·부가세

슈퍼마켓/편의점 등은 품목 상세내역이 있어야 적격 증빙으로 인정되는데 이를 정산 시점에 놓치기 쉽고, 영수증의 부가세 정보가 정산 시스템에 제대로 반영되지 않는 경우가 있다.

## 3. 해결 방안 (To-Be)

### 3.1 증빙 자동화 — "찍으면 끝"

영수증 촬영 한 장으로 원본은 보존하고 크롭본을 자동 생성, Vision LLM으로 가맹점·금액·통화·일시·품목·부가세를 구조화 추출한다. 크롭이 실패해도 원본을 그대로 쓸 수 있어 흐름이 끊기지 않는다. (상세: [`sot/02_architecture_db_schema.md`](../sot/02_architecture_db_schema.md))

### 3.2 정산단위 — 정산 규정의 배포 단위

이번 고도화의 핵심 아이디어는 "출장비 한도"와 "AI구독료 같은 특수정산"이 사실은 같은 종류의 문제라는 점이다. 둘 다 **① 적용 대상 비목 ② 한도 ③ 전결라인 ④ 적요 형식**이 미리 정해진 규정이고, 다만 누가 언제 만드느냐만 다르다.

이걸 **정산단위**이라는 하나의 엔티티로 통합했다:

- 관리자가 특수정산 정산단위를 작성하면 → 저장 즉시 시스템에 배포되어, 메일로 안내하고 사람이 기억할 필요 없이 다음 영수증 처리부터 바로 적용된다.
- 직원이 "출장모드 시작"을 누르면 → 출장 정산단위가 자동 생성되어 동일한 방식으로 적용된다.
- 영수증을 촬영하면 시스템이 맞을 법한 정산단위를 추천하고, 직원은 그중에서 고르기만 하면 비목·한도·전결라인·적요가 전부 자동으로 붙는다.

즉 **"규정을 배포하면 정산이 알아서 따라간다"** — 새로운 정산 유형이 생겨도 코드를 새로 짤 필요 없이 정산단위 하나를 추가하면 된다. (상세 스키마: [`sot/02_architecture_db_schema.md`](../sot/02_architecture_db_schema.md), 시나리오: [`sot/03_user_scenarios.md`](../sot/03_user_scenarios.md))

### 3.3 적격증빙·부가세 보조 확인 (Optional)

편의점/마트 등 상세내역이 필요한 결제는 증빙 누락 시 경고를 띄우고, 부가세는 추출값을 화면에서 바로 확인·수정할 수 있게 한다. 상신 자체를 막지는 않는다 — 실수를 줄이는 보조 장치이지, 통제 장치가 아니다.

## 4. 기대 효과

| 효과 | 설명 |
|---|---|
| 정산 소요시간 단축 | 8단계 수작업(촬영-크롭-첨부-전결 확인)이 촬영 한 번 + 확인 몇 번으로 축소 |
| 신규 정산유형 대응 속도 | 특수 예산이 새로 생겨도 개발 없이 정산단위 하나 배포로 즉시 대응 — 기존엔 메일 안내 후 사람이 기억해서 처리 |
| 정산 규정 준수 오류 감소 | 전결라인·한도·비목이 정산단위에서 자동으로 붙어 "사람이 규정을 깜빡함"으로 인한 오류가 줄어듦 |
| 실수 조기 발견 | 적격증빙 누락, 부가세 불일치를 정산 시점에 바로 경고 — 사후 반려·재작업 감소 |
| 확장 가능한 구조 | 이번엔 Mock e-accounting이지만, 실제 e-accounting과 연동해도 "정산단위 배포 → 자동 반영"이라는 모델 자체는 그대로 재사용 가능 |

## 5. 데모로 보여주는 것 (요약)

1. 관리자가 "AI구독료" 정산단위를 배포한다.
2. 직원이 "출장모드 시작"만으로 출장 정산단위가 자동으로 생긴다.
3. 영수증을 찍으면 정산단위가 추천되고, 고르기만 하면 전표가 거의 완성된다.
4. 편의점 영수증에서는 적격증빙 경고가, AI구독 결제에서는 배포된 정산단위가 바로 인식되는 걸 보여준다.
5. 사람은 "확인하고 상신"만 한다.

(전체 3분 시나리오: [`sot/06_spec.md`](../sot/06_spec.md))
