# data_sample — 영수증 샘플 데이터

OCR·분류·매칭 개발과 데모에 쓸 **영수증 샘플과 그 설명**을 모아두는 폴더입니다.
팀원 누구나 참고할 수 있도록 저장소에 함께 올립니다.

## 구조

```
data_sample/
├── README.md          이 문서 (샘플 목록표 + 확인된 이슈)
├── images/            실물 영수증 사진 (촬영본) — 촬영·크롭·Vision OCR 경로
├── pdf/               전자 영수증·인보이스 (PDF) — 텍스트 파싱 경로
└── expected/          샘플별 OCR 기대 결과 JSON
```

`images/`와 `pdf/`를 나눈 이유: PDF는 텍스트 레이어가 이미 있어 Vision OCR 없이 파싱할 수 있습니다.
처리 분기가 다르므로 폴더로 구분합니다.

## 등록 방법

1. 실물 사진은 `images/`에, 전자 영수증 PDF는 `pdf/`에 넣습니다.
2. 아래 **샘플 목록** 표에 한 줄을 추가합니다.
3. `expected/<원본과 같은 이름>.json`에 기대 결과를 적습니다. 형식은 [docs/02-API-CONTRACT.md](../docs/02-API-CONTRACT.md)의 `Receipt.ocr`을 따르고, 계약에 없는 보조 항목은 `_` 접두사를 붙입니다 (`_tax`, `_approvalNo`, `_expectedCategory`, `_note`).

## ⚠️ 올리기 전 반드시 확인

[CONTRIBUTING.md](../CONTRIBUTING.md)의 "실제 영수증·법인카드 정보는 커밋하지 않습니다" 규칙이 적용됩니다.

- **카드번호는 가립니다.** 이미지 위에 검은 박스로 덮은 뒤 저장하세요.
- **PDF는 검은 박스로 덮는 것만으로 부족합니다.** 아래 글자가 텍스트로 그대로 남아 복사하면 읽힙니다. 텍스트 자체를 제거(redaction)해야 합니다.
- 마스킹 대상: **카드번호, 결제자 실명, 개인 이메일**. 이미지 영수증처럼 일부를 `*`로 표시합니다 (`YANG B****`, `ra****@gmail.com`, `Visa - ****`).
- **멤버십·포인트 정보는 마스킹이 아니라 삭제합니다.** 해피포인트·OK캐쉬백 등의 고객명, 포인트 카드번호, 적립/잔여 포인트 블록은 통째로 지웁니다. 정산에 쓰이지 않는 정보입니다.
- **주문내역 화면을 올릴 때는 배송정보 블록을 잘라냅니다.** 이름·휴대폰 번호·집 주소·공동현관 출입정보가 들어 있습니다. 상품정보와 결제정보만 남기면 충분합니다.
- 한 번 커밋하면 파일을 지워도 **git 이력에 남습니다.** 넣기 전에 확인하세요.
- 가맹점명·금액·일시·품목은 OCR 개발에 필요하므로 그대로 둡니다.
- 이미지 한 장은 **2MB 이하**로 리사이즈합니다. (CONTRIBUTING: 대용량 바이너리 커밋 금지)
- 확실하지 않으면 올리기 전에 팀에 먼저 물어봅니다.

## 분류 규칙 — 출장비 판정 (2026-07-22 확정)

**가맹점만으로는 계정과목을 정할 수 없습니다.** 판정 기준은 장소가 아니라 **출장 기간**입니다.

### 규칙 0 — 그 여정이 출장인지부터 판정한다

근무지 기준으로 **두 조건을 모두** 만족해야 출장입니다. 하나라도 못 채우면 **외근**이고, 외근은 출장비가 아닙니다.

| 조건 | 기준 |
|---|---|
| 거리 | **네이버지도 길찾기 편도 30km 이상** |
| 행정구역 | **행정구역이 바뀔 것** |

둘은 AND입니다. 행정구역만 바뀌고 가까우면 외근, 멀어도 같은 행정구역이면 외근입니다.

| 구분 | 계정과목 |
|---|---|
| 출장 (두 조건 충족) | `TRAVEL_*` — 아래 규칙 1·2 적용 |
| **외근** (조건 미충족) | `SNACK` 등 **일반 계정. 출장비 아님** |

근무지는 **서울 종로**입니다 (영수증에 나온 회사 주소 SK서린빌딩·종로타워, [fixtures/transactions.json](../fixtures/transactions.json)의 소속 Upstream기술팀).

### 규칙 1 — 출장 기간 안이면 출장비

출장 기간에 포함되는 날의 사용 건은 전부 출장비입니다. 출장 시작일과 종료일도 포함합니다.

> 5/1–5/3 출장(2박 3일)이면 **5/1, 5/2, 5/3 사용 건이 모두 출장비**입니다.

장소는 따지지 않습니다. 출장 첫날 출발 전 근무지 근처에서 산 커피도 출장 기간 안이므로 출장비입니다.

**출장 식대에는 간식·카페·식당 결제가 모두 포함됩니다.** 업종(커피숍/편의점/일반음식점)으로 나누지 않고, 출장 기간이면 전부 `TRAVEL_MEAL`입니다.

| 출장 기간 안 | 출장 기간 밖 |
|---|---|
| 식당·카페·편의점 → `TRAVEL_MEAL` | 카페·편의점 → `SNACK` |
| 교통 → `TRAVEL_TRANSPORT` | |
| 숙박 → `TRAVEL_LODGING` | |

### 규칙 2 — 기간 판정에 쓰는 날짜는 결제일이 아닐 수 있다

**KTX·항공·숙박은 사전 결제합니다.** 결제일로 판정하면 출장 기간을 벗어나 오분류됩니다.

| 결제 성격 | 기간 판정에 쓰는 날짜 |
|---|---|
| 일반 결제 (식당·카페·편의점) | 결제일시 |
| **사전 결제 (철도·항공)** | **탑승일** |
| **사전 결제 (숙박)** | **투숙일** |

여기서 날짜가 **두 가지 용도로 갈립니다.** 하나로 합치면 안 됩니다.

| 용도 | 쓰는 날짜 | 이유 |
|---|---|---|
| 카드승인내역 매칭 | **결제일시** (`paidAt`) | 카드사 승인 기록과 맞춰야 하므로 |
| 출장 기간 판정 | **이용일** (`serviceDate`) | 실제로 언제 쓴 건지가 기준이므로 |

→ [02-API-CONTRACT.md](../docs/02-API-CONTRACT.md)의 `Receipt`에 **`serviceDate` 필드 추가가 필요합니다.** 현재는 `paidAt` 하나뿐이라 이 규칙을 구현할 수 없습니다.

`expected/*.json` 표기:

```jsonc
"paidAt": "2026-04-09T00:00:00+09:00",   // 승인일 — 카드 매칭용
"_serviceDate": "2026-04-15",            // 탑승일 — 출장 기간 판정용
"_tripDateBasis": "serviceDate",         // 이 건은 어느 날짜로 판정하는지
"_tripId": "trip_2026-04-15"
```

### ⚠️ 파리크라상 건 — 규칙 0과 규칙 1이 충돌합니다

| 규칙 | 판정 |
|---|---|
| 규칙 0 (거리) | 종로 → 서울역 약 4km. **30km 미달 → 외근 → `SNACK`** |
| 규칙 1 (기간) | 2026-04-15는 부산 출장일. **출장 기간 안 → `TRAVEL_MEAL`** |

결제 지점 자체는 출장 조건을 못 채우지만, **그날은 출장일**입니다. 07:47 결제 11분 뒤 부산행 KTX를 탔습니다.

규칙 0은 **여정이 출장인지**를 정하고, 규칙 1은 **출장 기간 내 결제를 어떻게 볼지**를 정합니다. 그렇게 읽으면 이 건은 `TRAVEL_MEAL`입니다. 규칙 0을 결제 건마다 적용하면 `SNACK`입니다.

**어느 쪽인지 확정이 필요합니다.** 출장 당일 출발 전 근무지 근처 결제는 실무에서 흔합니다. 현재는 `TRAVEL_MEAL`로 적어뒀습니다.

> **📌 화면 반영 예정** — 위 판정 기준(거리 30km·행정구역 변경, 출장 기간, 사전결제는 탑승일 기준)은 **중간 대시보드에 가이드라인으로 노출**합니다. 사용자가 왜 이 건이 출장비로 잡혔는지 화면에서 바로 확인할 수 있게 합니다. 화면 설계 시 반영할 것.

### 구현 시 문제

1. **거리 계산을 데모에서 어떻게 하는가** — 네이버지도 길찾기 API 연동은 10시간 안에 무리입니다. 지역별 거리 고정 테이블이나 광역시·도 단위 판정으로 대체하는 게 현실적입니다.
2. **가맹점 주소를 못 얻는 건이 있습니다** — KTX 전자영수증에 찍힌 주소는 한국철도공사 본사(대전)입니다. 실제 이동 구간(서울→부산)은 본문에서 따로 읽어야 합니다.
3. **행정구역 단위를 정해야 합니다** — 시·도인지, 시·군·구인지. 서울 종로구 → 용산구는 구 단위로는 변경이지만 시 단위로는 동일합니다.

⚠️ **갈 때는 부산, 올 때는 울산입니다.** 출장지를 한 곳으로 단정하면 귀환편이 안 맞습니다. `Trip`은 목적지 하나가 아니라 **경로**로 다뤄야 합니다.

### 결정이 필요한 사항

1. **출장 여부를 무엇으로 판정하는가** — `Trip` 등록 데이터를 근거로 할지, 영수증의 지역명으로 추정할지. 지금 [fixtures/](../fixtures/)에 **`trips.json`이 없어** 어느 쪽도 검증할 수 없습니다. 위 동선을 `trip_2026-04-15`로 만들어 두면 됩니다.
2. **출장 시작·종료 경계** — 출발지(서울역) 결제를 출장에 포함하기로 했습니다. 귀환 후 결제도 같은 기준을 적용할지 확인이 필요합니다.
3. **복수 지역 출장** — [02-API-CONTRACT.md](../docs/02-API-CONTRACT.md)의 `Trip`은 `destination` 하나뿐입니다. 부산·울산처럼 여러 곳을 도는 출장을 담을 수 없습니다.
4. **2025년 부산 건 2장** — 스토리웨이·플루토커피는 출장이었는지 확인이 필요합니다. 출장이면 `TRAVEL_MEAL`, 아니면 `SNACK`입니다.

## 환율 규칙 (데모용 — 2026-07-22 확정)

실제로는 E-Accounting이 환율을 계산해 반영하지만, 데모 개발 기간에는 **USD 1,500원 고정**으로 갑니다.

| 항목 | 규칙 |
|---|---|
| 적용 환율 | **USD 1,500원 / JPY 9.0원 고정** |
| 원본 통화 | **그대로 보존**한다. USD 결제는 `amount: 22.00`, `currency: "USD"`로 받고 입력받는다 |
| 원화 환산 | 저장 시점에 함께 계산해 **별도 필드로 저장**한다. 화면·전표·한도차감은 이 값을 쓴다 |
| 환산식 | `amountKRW = round(amount × fxRate)` |
| 문서상 환산액 | 참고용으로만 보관한다. 계산에는 쓰지 않는다 (아래 이슈 4번) |

적용 환율표 (데모 고정):

| 통화 | 환율 |
|---|---|
| USD | **1,500** |
| JPY | **9.0** |
| KRW | 1 |

DB에 함께 저장할 필드:

```jsonc
{
  "amount": 22.00,        // 원본 통화 금액 — 입력·표시용
  "currency": "USD",      // 원본 통화
  "fxRate": 1500,         // 적용 환율 (KRW 기준)
  "amountKRW": 33000      // 환산값 — 집계·한도차감·전표용
}
```

`amount`만 저장하면 나중에 환율이 바뀔 때 과거 건이 소급 변경됩니다. **결제 시점 환율을 건마다 같이 저장**해야 값이 고정됩니다.

이 규칙을 적용한 환산 결과:

| 샘플 | 원본 | 환산 |
|---|---|---|
| ChatGPT 구독 | $22.00 | ₩33,000 (×1,500) |
| Claude 구독 | $110.00 | ₩165,000 (×1,500) |
| 나머지 6건 | KRW | 환산 불필요 (`fxRate: 1`, `amountKRW = amount`) |

> JPY 9.0은 [fixtures/fx.json](../fixtures/fx.json)의 기존 값과 같습니다. USD만 1380 → 1500으로 고쳐야 합니다.
> v1.1에서 `tx_101~108` 도쿄 출장(JPY) 건이 추가됐으므로 JPY 환율도 확정해 둡니다.

## 부가세 규칙 — 국내는 분리, 해외는 합산 (2026-07-22 확정)

부가세는 분리해서 관리하되, **국내 결제와 해외 결제를 다르게 처리**합니다.

| 구분 | 처리 | 필드 |
|---|---|---|
| **국내 결제** | 공급가액과 부가세를 **분리** | `supplyKRW` + `vatKRW` |
| **해외 결제** | 부가세를 **합산**해 총액으로 관리 | `supplyKRW` = 총액, `vatKRW` = 0 |

해외 사업자가 청구한 부가세는 국내 매입세액 공제 대상이 아니므로 분리할 실익이 없습니다. 분리해서 넣으면 부가세 신고 집계가 틀어집니다.

```jsonc
// 국내 — 백세촌 국밥 12,000원
"_tax": { "supplyKRW": 10909, "vatKRW": 1091, "taxFreeKRW": 0 }

// 해외 — ChatGPT $22.00 → ₩33,000
"_tax": {
  "supplyKRW": 33000,          // 부가세 합산한 총액
  "vatKRW": 0,                 // 분리하지 않음
  "overseasCombined": true,    // 해외 합산 처리 표시
  "documentVatUSD": 2.00       // 문서상 부가세 — 참고용
}
```

필드명은 v1.1에서 확정된 [02-API-CONTRACT.md](../docs/02-API-CONTRACT.md)의 `supplyKRW`/`vatKRW`를 따릅니다.

### 부가세 표기가 없는 샘플

영수증에 부가세 내역이 없는 건이 있습니다. 임의로 10%를 역산하지 않고 `vatKRW: null`로 두었습니다.

| 샘플 | 상태 |
|---|---|
| 우체국 등기 | 우편료. 부가세 표기 없음 |
| 플루토커피 | 합계만 표기 |
| KTX 2건 | 결제금액만 표기 |

→ 부가세 미표기 건을 **역산할지, null로 둘지** 팀 결정이 필요합니다. 역산하면 실제 면세 품목에서 틀린 값이 들어갑니다.

## 이 샘플들로 확인된 이슈

팀 결정이 필요한 사항입니다. 임의로 처리하지 않았습니다.

1. **계정과목 코드 신설 필요** — 우편료는 **판관비**로 처리하기로 정했습니다(2026-07-22). 그런데 [02-API-CONTRACT.md](../docs/02-API-CONTRACT.md)의 코드표는 TRAVEL/SNACK/WELFARE 뿐이라 담을 코드가 없습니다. `SGA_POSTAGE`를 임시로 쓰고 있으니 코드표에 추가해야 합니다. 계약 문서 변경이므로 **전원 합의 후 문서 먼저 수정**이 필요합니다.
2. **예매 건의 paidAt** — KTX는 **승인일(2026-04-09)과 이용일(2026-04-15)이 6일 차이**입니다. 영수증 본문의 큰 날짜(이용일)를 paidAt으로 잡으면 매칭 규칙의 "일시 근접(±24h)" 점수를 못 받아 자동매칭이 실패합니다. 항공·철도·숙박 등 예매성 결제는 **승인일자**를 써야 합니다.
3. **인보이스와 영수증 중복** — 같은 결제 건에 문서가 2장씩, 총 **2세트**(ChatGPT·Claude) 있습니다. `invoiceNo` 기준 dedupe 규칙이 없으면 전표가 2배로 생성됩니다. 구분 단서: 인보이스는 `Amount due` + `Pay online`, 영수증은 `Amount paid` + `Date paid` + Payment history 표.
4. **환율 값이 세 갈래로 갈립니다** — ChatGPT 영수증의 문서상 환산액은 약 **1,511원/USD**(VAT $2.00 = ₩3,022), [fixtures/fx.json](../fixtures/fx.json)은 **1,380원**, 위에서 정한 데모 규칙은 **1,500원**입니다. Claude 영수증에는 환산액이 아예 없습니다. **시스템 환율(1,500)을 쓰고 문서상 환산액은 참고용으로만 보관**하기로 정했습니다. → `fixtures/fx.json`의 USD를 1380 → 1500으로 고쳐야 합니다 (아래 "다른 담당자에게 요청할 사항" 참고).
5. **소수점 금액** — USD 결제는 `22.00`처럼 소수점이 있습니다. 원화 정수만 가정한 코드에서 깨질 수 있습니다.
6. **paidAt 판정 규칙** — 계산일시와 인쇄/출력일시가 다른 영수증이 있습니다(우체국 1분차, 플루토커피 16초차). 계산·접수 일시를 채택하기로 임시 정했습니다.
7. **cardLast4를 대부분 못 얻음** — 8건 중 뒤 4자리가 나오는 건 ChatGPT 영수증(9112) 하나뿐입니다. 매칭이 카드번호에 의존하면 대부분 실패합니다.
8. **fixtures와 안 맞음** — [fixtures/transactions.json](../fixtures/transactions.json)의 카드승인내역 7건은 여기 샘플 8건과 **한 건도 대응되지 않습니다**. 매칭 데모를 하려면 샘플에 맞는 승인내역을 추가해야 합니다.

### ⚠️ 복지비 한도가 모자랍니다

[fixtures/budgets.json](../fixtures/budgets.json)의 `WELFARE_AI` 잔여 한도는 **₩180,000**입니다.
그런데 AI구독 샘플 2건의 환산 합계는 **₩198,000**(33,000 + 165,000)으로 **₩18,000 초과**입니다.

환율을 1,380원으로 계산해도 ₩182,160이라 여전히 초과입니다. 데모 2:40의 "복지비 잔여한도 자동 차감" 장면을 그대로 시연하면 한도 초과 상태가 됩니다. 셋 중 하나를 정해야 합니다.

1. `WELFARE_AI` 한도를 올린다 (예: 30만 → 40만)
2. 데모에는 ChatGPT 건($22 → ₩33,000)만 쓴다
3. **한도 초과 시나리오를 의도적으로 시연한다** — 초과 경고 UI를 보여주는 쪽이 오히려 데모에 설득력이 있을 수 있습니다
