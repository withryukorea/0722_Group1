# 3. User Scenarios

## 직원 시나리오

### E1. 출장모드 시작 (TRIP 정산단위 생성)

1. 직원이 앱에서 "출장모드 시작" 선택.
2. 목적지/기간/인원 입력.
3. 앱이 `POST /api/presets`(`type=TRIP`) 호출 — 일일 Cap = 인당한도 × 인원으로 계산되어 `rules.limitKRW`에 반영.
4. 생성된 정산단위가 즉시 활성화되고, 잔여한도 화면에 표시됨.

Acceptance:

- 품의 시스템 연동 없이 앱에서 즉시 정산단위가 생성된다 (mock).
- 기간이 겹치는 다른 TRIP 정산단위와의 충돌 처리는 이번 PoC 범위 밖 (단일 활성 출장 가정).

### E2. 영수증 촬영 및 정산단위·비목 확정

1. 직원이 영수증 촬영.
2. 서버가 원본 보존 + 크롭 + OCR 수행.
3. 리뷰 화면에 OCR 결과 표시, 자동추천 정산단위가 하이라이트됨.
4. 직원이 정산단위 확정 — 추천을 그대로 두거나, 다른 활성 정산단위, 또는 "일반 결제(정산단위 없음)"로 변경 가능.
5. 선택한 정산단위의 `allowedAccountCodes`가 2개 이상이면 비목 선택 칩 노출 → 직원이 선택 (1개면 자동 세팅, 이 단계 생략).
6. 부가세 추출값 표시 → 직원이 확인 또는 수정 (Optional 기능).
7. 적격증빙이 필요한 계정과목(편의점/마트)이면 경고 배지 노출 — 상신은 막지 않음 (Optional 기능).
8. 직원이 "다음" → 영수증 저장.

Acceptance:

- 정산단위 자동추천은 참고용이며, 최종 확정은 항상 사용자가 한다.
- 비목 선택은 선택된 정산단위의 `allowedAccountCodes`로 제한된다.
- 적격증빙 경고가 떠도 다음 단계로 진행할 수 있다 (하드 블록 없음).
- 부가세 수정 이력은 `vat.confirmed`에 저장된다.

### E3. 정산하기 (거래 매칭)

1. 직원이 "정산하기" 탭 진입.
2. 저장된 영수증들과 카드승인내역이 자동 매칭 (기존 로직 그대로, 정산단위 지정과 무관).
3. 매칭 스코어가 낮은 건은 사용자가 후보 중 확인.

Acceptance:

- 거래 매칭은 정산단위 지정 여부와 완전히 독립적으로 동작한다.

### E4. 전표 생성 및 상신

1. 매칭된 건들로 "전표 생성".
2. 정산단위가 지정된 건은 그 정산단위의 계정과목(사용자가 고른 비목)·전결라인·적요 템플릿을 사용.
3. 정산단위가 없는 건은 기존 자동분류 + `approval-rules.json` fallback 사용.
4. 직원이 검토 후 "상신".
5. 사용된 정산단위의 `usage`가 차감 갱신된다.

Acceptance:

- 전표 상신 후 관리자 콘솔에 즉시 노출된다 (3초 폴링).
- 정산단위 한도를 초과해도 상신은 가능하다 (경고만, 하드 블록 없음).

## 관리자 시나리오

### A1. 정산단위 배포

1. 관리자가 관리자 콘솔의 "정산단위" 섹션에서 "새 정산단위" 작성.
2. 이름 / 유형(RECURRING·CAMPAIGN) / 허용 비목 / 한도 / 전결라인 / 적요 템플릿 / 매칭 키워드 입력.
3. 저장 즉시 활성 정산단위 목록에 노출 — 대상 사용자(`u_me`)의 다음 영수증 리뷰부터 후보로 뜬다.

Acceptance:

- 메일 발송 등 별도 안내 단계 없음 — 저장이 곧 배포다.
- 관리자는 기존 정산단위를 비활성화하거나 수정할 수 있다.

### A2. 카드내역·전표 확인 (기존 기능, 변경 없음)

기존 `admin.js`/`index.html` 그대로 — 카드내역 리스트, 접수전표+결재선 확인, 데모 리셋 버튼.
