# 3. User Scenarios

> 원칙: **유입(증빙 수집)만 모바일, 정산·상신은 이어카운팅.** 모바일에는 정산단위 생성 기능이 없다.

## 직원 시나리오

### E1. 출장 정산단위 확보 (모바일 아님 — 이어카운팅/품의)

**해외출장** — 품의 승인 시 시스템이 TRIP 정산단위를 자동 생성한다. 품의 시스템 연동은 비범위이므로 PoC에서는 **seed로 시뮬레이트** (데모 시작 시 "7월 도쿄 출장" 정산단위가 이미 존재).

**국내출장** — 품의가 애매한 경우, 직원이 **이어카운팅**의 정산단위 화면에서 직접 생성:

1. 목적지·기간만 입력.
2. 일비·허용비목·전결라인은 **국내출장 표준 규정 기본값이 자동** 세팅.
3. 생성 즉시 활성화 — 모바일 리뷰 화면의 후보로 노출된다.

Acceptance:

- 모바일 앱에서는 정산단위를 생성할 수 없다 (선택만 가능).
- 기간이 겹치는 다른 TRIP 정산단위와의 충돌 처리는 이번 PoC 범위 밖 (단일 활성 출장 가정).

### E2. 영수증 촬영 (모바일 · 7단계 유입 흐름)

1. 직원이 영수증 촬영 → 원본파일 서버 전송.
2. 서버가 자동 크롭 (원본은 항상 보존).
3. **크롭 이미지 확인** — 어긋나면 "다시 크롭", 실패 시 원본 사용.
4. 서버가 정보 파싱 (OCR).
5. **파싱 확인 + 정산단위·비목 선택**: 자동추천 정산단위가 하이라이트되고, 직원이 확정 — 추천 그대로 / 다른 활성 정산단위 / "일반 결제(정산단위 없음)".
6. 선택한 정산단위의 `allowedAccountCodes`가 2개 이상이면 비목 선택 칩 노출 (1개면 자동 세팅·생략). 부가세 확인/수정, 적격증빙 경고(Optional)도 이 화면에 표시.
7. 저장 → 보관함. **정산은 여기서 하지 않는다** — 이어카운팅에서 이어짐(E4).

Acceptance:

- 사용자 개입은 크롭 확인(3)과 파싱 확인·선택(5~6) 두 지점뿐이다.
- 정산단위 자동추천은 참고용이며, 최종 확정은 항상 사용자가 한다.
- 비목 선택은 선택된 정산단위의 `allowedAccountCodes`로 제한된다.
- 적격증빙 경고가 떠도 진행할 수 있다 (하드 블록 없음).
- 부가세 수정값은 `vat.confirmed`에 저장된다.

### E3. PC 영수증 업로드 (이어카운팅 · 비출장/구독형)

AI구독료처럼 PDF 인보이스로 오는 건:

1. 직원이 이어카운팅의 영수증 업로드 화면에서 파일(PDF/이미지) 업로드.
2. 파싱 후 정산단위가 키워드로 자동추천됨 (예: "ANTHROPIC" → AI구독 정산단위).
3. 확인 후 정산(E4)으로 이동.

### E4. 매칭·정산·상신 (이어카운팅)

1. **매칭 화면**: 업로드된 영수증들과 카드승인내역이 자동 매칭 (금액/일시/가맹점 스코어링, 정산단위 지정과 무관). 낮은 점수 건만 후보 중 수동 확정. 행별 비목 확인, 증빙 추가·해제 가능.
2. **정산(전표작성) 화면**: 유형에 맞는 기존 이어카운팅 화면 사용 — 해외출장이면 "해외출장비 정산", 비출장이면 "법인카드 정산(현업완결)".
3. 정산단위가 지정한 칸은 **자동완성·고정**: 계정과목(비목) · 예산부서(Cost Center) · 적요(양식→자동 생성, 예: "정성훈팀원_7월 AI Frontier") · 결재선(양식→자동 전개: 기안[기안자]→검토(고정)→승인[차상위자]).
4. 나머지(금액·통화·환율·부가세·증빙일자 등)는 일반 정산처럼 확인/입력. 모바일에서 찍은 영수증은 증빙으로 자동 첨부.
5. 직원이 검토 후 "상신" → 사용된 정산단위의 `usage` 차감 갱신.

Acceptance:

- 정산단위 지정 건은 자동완성 칸에 손댈 필요가 없다 (수정은 가능하되 기본값이 채워져 있음).
- 정산단위가 없는 건은 기존 자동분류 + `approval-rules.json` fallback 사용.
- 전표 상신 후 문서함/접수 화면에 즉시 노출된다.
- 정산단위 한도를 초과해도 상신은 가능하다 (경고만, 하드 블록 없음).

### E5. 정산단위 대시보드 조회 (모바일 · 읽기 전용)

1. 직원이 모바일 "내 정산 현황"에서 배정된 정산단위 목록 확인.
2. **TRIP**: 일별 구분 없이 비목별 누적 + **일당(per-diem)** 한도 대비 사용/잔여 중심.
3. **RECURRING**: 월 한도 대비 사용/잔여 게이지.
4. 상신이 완료되면 사용량이 자동 반영되어 갱신된다.

## 관리자 시나리오

### A1. 정산단위 배포 (이어카운팅)

1. 관리자가 이어카운팅 정산단위 관리 화면에서 "새 정산단위" 작성.
2. 입력: 이름 / 유형(RECURRING·CAMPAIGN) / 허용 비목(실계정코드) / **Cost Center** / 한도·기간 / **전결라인 양식**(기안[기안자]→검토 지정→승인[차상위자]) / **적요양식**([이름][직급]_[월]월 …) / 매칭 키워드.
3. 저장 즉시 활성 목록에 노출 — 대상 사용자의 다음 영수증 처리부터 후보로 뜬다.

Acceptance:

- 메일 발송 등 별도 안내 단계 없음 — 저장이 곧 배포다.
- 관리자는 기존 정산단위를 비활성화하거나 수정할 수 있다.

### A2. 카드내역·전표 확인 (기존 기능, 변경 없음)

기존 이어카운팅 화면 그대로 — 카드내역 리스트, 접수전표+결재선 확인(`mydocs-all` 등), 데모 리셋.

## 데모 케이스 매핑

상세 목업은 `demo/demo-cases.html`(PDF 동봉) 참조:

- **CASE 1 해외출장(도쿄, TRIP)**: seed 정산단위 → E2(모바일 촬영) → E4(해외출장비 정산) → E5(대시보드)
- **CASE 2 비출장(AI구독, RECURRING)**: A1(관리자 배포) → E3(PC 업로드) → E4(법인카드 정산·현업완결)
