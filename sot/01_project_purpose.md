# 1. Project Purpose

## Project Name

0722_Group1 — "찍으면 끝" 법인카드 정산 자동화 (고도화 PoC: 정산단위 기반 정산엔진)

## Objective

법인카드 정산 프로세스를 효율화하는 PoC. 영수증 촬영/업로드만으로 크롭·OCR 파싱·카드내역 매칭이 이루어지고, **정산단위**라는 단일 개념으로 정산 규정(비목·Cost Center·적요양식·전결라인·한도)을 배포하면, 이어카운팅 정산 화면의 지정 칸이 자동으로 채워져 사용자가 손댈 부분이 최소화된다. 적격증빙 검토와 부가세 확인은 보조(Optional) 기능으로 포함한다.

역할 원칙: **유입(증빙 수집)만 모바일, 정산·상신은 이어카운팅.**

## Target Users

| User | Description |
|---|---|
| 직원 (`u_me`, 단일 하드코딩 사용자) | 모바일: 영수증 촬영·크롭/파싱 확인·정산단위/비목 선택·대시보드 조회. 이어카운팅: PC 업로드, 국내출장 정산단위 생성, 매칭 확인, 정산·상신 |
| 관리자 (이어카운팅 운영자) | 정산단위 작성·배포, 카드내역/접수전표 확인 — 실제 회계팀 역할을 시연 |
| (참고, 비범위) 실제 회계팀/전결자 | Mock 밖의 실제 결재·회계 처리. 결재라인 "표시"까지만 다룸 |

## Business Context

- 사내망 연결이 불가능한 환경에서 개발·시연하기 위해 실제 e-accounting 시스템을 Mock REST 서버 + 관리자 웹화면으로 대체.
- 최초 산출물: 2026-07-22, 10시간·5인 해커톤 — 촬영→크롭→OCR→매칭→전표 MVP (`docs/01-PLANNING.md`).
- 이번 라운드: 고도화 PoC 기획 — 출장모드·복지비를 정산단위로 통합 재설계, 정산단위 배포 체계 추가, 적격증빙·부가세 보조검증 추가.
- 통화: KRW 기준, 고정 환율 테이블로 주요 통화 환산 (실시간 API 아님).
- 회계 특이사항: 슈퍼마켓/편의점 등은 품목 상세내역이 있어야 적격 증빙, 부가세는 별도 확인이 필요.

## Expected Outputs

- 배포된 정산단위 목록 (관리자 배포 / 직원 국내출장 생성 / 해외출장 품의 자동생성[seed]).
- 원본/크롭 분리 저장된 영수증 이미지, OCR 결과(JSON, 부가세 포함).
- 정산단위·비목이 확정된 영수증, 카드내역과의 매칭 결과.
- 자동 생성된 전표(계정과목·전결라인·적요 포함) 및 상신 상태.
- 적격증빙 경고, 부가세 확인/수정 이력.
- 이어카운팅에서 위 전체 상태(접수 전표·결재선·정산단위 사용량) 확인.

## Current Stage

2026-07-22 기준:

- 해커톤 MVP 서버(Mock E-Accounting + seed fixtures) 구현됨 — `server/`, `fixtures/`.
- 정산단위 엔진 백엔드 구현됨 — `server/routes/presets.js`, `fixtures/presets.json` (Trip/Budget 흡수 완료, `trips.js` 삭제).
- 이어카운팅 화면 골격(`eaccounting/`)·모바일 앱 골격(`app/`, `design/`) 존재 — 디자인 위주 진행 중.
- 데모 케이스 확정: 유형별 2케이스 (`demo/demo-cases.html` 참조).

## Success Criteria

- 정산단위를 배포하면, 그 규칙(비목·Cost Center·적요양식·전결라인·한도)이 코드 변경 없이 이어카운팅 정산 화면에 자동 반영된다.
- 영수증 유입(촬영/업로드)부터 상신까지 사람은 "확인·선택"만 한다 — 정산단위 지정칸은 무입력.
- 출장비(TRIP)와 특수정산(AI구독 등, RECURRING/CAMPAIGN)이 동일한 정산단위 메커니즘으로 동작함을 데모로 보여준다.
- 적격증빙 경고·부가세 확인 기능이 최소한 동작한다 (Optional이지만 포함이 목표).

## Non-Goals

- 실제 사내망/실제 e-accounting 시스템 연동.
- 실제 메일 발송 (정산단위는 시스템이 즉시 배정, 메일 안내 단계 mock조차 없음).
- 다중 사용자, 로그인/권한 관리 (단일 사용자 하드코딩 유지).
- 품목별 비목 분리 (영수증 1건 = 비목 1개. 섞인 품목은 사용자가 결제를 나누는 것을 전제).
- 상신 자체를 막는 하드 블록 (적격증빙·부가세는 경고/확인 수준).
- 실시간 환율 연동, 실제 결재선 진행(승인 클릭 등).
- 프로덕션급 보안·백업·복구·거버넌스 체계 (레퍼런스 프로젝트 `Corp_Card_Receipt_Mgmt`의 모델은 이 PoC 스케일 밖).
