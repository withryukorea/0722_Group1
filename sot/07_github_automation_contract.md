# 7. 협업 계약 (경량 버전)

- 최종 갱신: 2026-07-22
- 범위: 이 프로젝트의 브랜치·머지·사람 판단 규칙

## 왜 레퍼런스와 다른가

`Corp_Card_Receipt_Mgmt`의 07번 문서는 GitHub Issue 라벨(`br:*`) 기반 자동화 워커 체계와 ADR 게이트를 정의한다. 이 프로젝트는 GitHub Actions/라벨 자동화를 쓰지 않는 5인 규모의 PoC이므로, 그 체계를 그대로 가져오지 않는다. 대신 `docs/03-ROLES-TIMELINE.md`에 있던 "통합이 안 터지게 하는 규칙"을 이 프로젝트의 협업 계약으로 그대로 승격한다.

## 핵심 규칙

1. **계약 우선**: API/데이터 스키마(Preset, Receipt 등)를 바꾸려면 `sot/02_architecture_db_schema.md`·`sot/05_api.md`를 먼저 고치고 전원에게 공지한다.
2. **fixtures가 진실**: 모든 모듈은 fixtures 입출력만으로 단독 실행/테스트 가능해야 한다.
3. **main은 항상 데모 가능**: 기능 브랜치에서 작업하고, 동작 확인 후에만 머지한다. 통합 단계(H6 이후에 해당하는 시점) 이후의 머지는 PM 역할 승인을 받는다.

## 사람 판단이 필요한 경우 (ADR 급)

아래는 "일단 진행하지 말고 먼저 확인" 대상이다:

- Preset 스키마 자체의 구조 변경 (필드 추가/삭제 수준을 넘어서는 재설계)
- Trip/Budget 흡수 결정을 되돌리는 변경 (다시 별도 엔티티로 분리하는 등)
- 적격증빙/부가세를 경고에서 하드 블록으로 바꾸는 변경 (Non-Goals 위반)
- 실제 사내망·실제 메일 발송·다중 사용자 인증 등 비범위 항목을 범위에 넣는 변경

이 경우 무엇을 바꾸려는지, 왜 필요한지, 되돌릴 수 있는지를 먼저 팀에 공유하고 합의한 뒤 진행한다.

## 세션 종료 시 기록할 것

정식 이슈 트래커가 없으므로, 큰 변경을 했다면 커밋 메시지 또는 PR 설명에 다음을 남긴다:

- 바뀐 파일
- 왜 바꿨는지 (어떤 SoT 결정을 반영했는지)
- 남은 작업 / 다음에 이어서 할 지점
